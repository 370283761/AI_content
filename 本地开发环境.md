## 快速启动

### 1. 克隆仓库

```bash
git clone <repo-url>
cd AI_content
```

### 2. 安装依赖

```bash
pnpm install
```

### 3. 配置环境变量

```bash
cp .env.example .env
```

编辑 `.env`，填入以下关键配置：

```bash
# 数据库
DATABASE_URL=postgresql://postgres:password@localhost:5432/ai_content

# Redis
REDIS_URL=redis://localhost:6379

# 对象存储（MinIO）
S3_ENDPOINT=http://localhost:9000
S3_ACCESS_KEY=minioadmin
S3_SECRET_KEY=minioadmin
S3_REGION=us-east-1

# 应用
NODE_ENV=development
APP_URL=http://localhost:3000
PROJECT_QUOTA_BYTES=1073741824      # 1GB
```

### 4. 启动依赖服务

```bash
docker compose up -d
```

服务启动后：
- **Postgres**：`localhost:5432`
- **Redis**：`localhost:6379`
- **MinIO**：
  - API: `localhost:9000`
  - Console: `localhost:9001`（用户名/密码：`minioadmin` / `minioadmin`）

### 5. 数据库迁移

```bash
pnpm prisma:generate
pnpm prisma:migrate
pnpm prisma:seed
```

### 6. 启动开发服务器

```bash
pnpm dev
```

访问：`http://localhost:3000`

## 确认工程启动正常

> M1 的 `/api/health` 会同时检查 Postgres、Redis 与 MinIO。任一依赖未启动时返回 `503` 和 `degraded`，但落地页仍可用于检查前端构建。

### 1. 检查依赖服务

```bash
# 检查 Docker 容器状态
docker compose ps

# 应显示三个服务均为 "Up"：
# - postgres
# - redis
# - minio
```

### 2. 检查数据库连接

```bash
# 方式一：通过 Prisma Studio
pnpm prisma studio

# 方式二：通过 psql
psql postgresql://postgres:password@localhost:5432/ai_content -c "SELECT 1;"
```

### 3. 检查 Redis 连接

```bash
redis-cli -h localhost -p 6379 ping
# 应返回：PONG
```

### 4. 检查 MinIO

访问 `http://localhost:9001`，使用 `minioadmin` / `minioadmin` 登录。

### 5. 检查 Next.js 开发服务器

访问 `http://localhost:3000`，应看到落地页或项目列表页。

### 6. 检查 API 可用性

```bash
curl http://localhost:3000/api/health
# 应返回：{"status":"ok"}
```

## 常见问题

### 问题 1：端口被占用

**症状**：`Error: listen EADDRINUSE: address already in use :::3000`

**解决**：

```bash
# macOS/Linux
lsof -ti:3000 | xargs kill -9

# Windows
netstat -ano | findstr :3000
taskkill /PID <PID> /F
```

### 问题 2：数据库连接失败

**症状**：`Error: Can't reach database server at localhost:5432`

**解决**：

1. 确认 Docker 服务正常：
   ```bash
   docker compose ps
   ```

2. 检查 `.env` 中的 `DATABASE_URL` 是否正确。

3. 重启 Postgres：
   ```bash
   docker compose restart postgres
   ```

### 问题 3：Prisma Client 未生成

**症状**：`Cannot find module '@prisma/client'`

**解决**：

```bash
pnpm prisma generate
```

### 问题 4：MinIO 连接失败

**症状**：上传文件时报错 `S3 connection refused`

**解决**：

1. 确认 MinIO 服务运行：
   ```bash
   docker compose ps minio
   ```

2. 检查 `.env` 中的 `S3_ENDPOINT`、`S3_ACCESS_KEY`、`S3_SECRET_KEY`。

3. 访问 `http://localhost:9001`，手动创建桶：
   - `references`
   - `results`
   - `exports`

### 问题 5：AI 任务超时

**症状**：AI 生成任务一直处于 `pending` 状态

**解决**：

1. 检查 `.env` 中的 `CLAUDE_API_KEY` 和 `GPT56_API_KEY` 是否正确。

2. 查看 BullMQ 队列日志：
   ```bash
   docker compose logs redis
   ```

3. 检查网关日志：
   ```bash
   # 前端控制台或服务端日志中搜索 "AITask"
   ```

### 问题 6：TypeScript 类型错误

**症状**：IDE 提示类型错误但代码可运行

**解决**：

```bash
# 重新生成 Prisma Client
pnpm prisma generate

# 重启 TypeScript 服务器（VSCode: Cmd+Shift+P > "Restart TS Server"）
```

## 日志与调试

### 查看应用日志

开发模式下，日志直接输出到控制台。

### 查看数据库查询日志

在 `prisma/schema.prisma` 中添加：

```prisma
generator client {
  provider = "prisma-client-js"
  log      = ["query", "info", "warn", "error"]
}
```

然后重新生成：

```bash
pnpm prisma generate
```

### 查看队列任务

使用 BullMQ Board（可选）：

```bash
pnpm add -D bull-board
# 在开发服务器中配置 Bull Board UI
```

或直接查询 Redis：

```bash
redis-cli -h localhost -p 6379
KEYS *
```
